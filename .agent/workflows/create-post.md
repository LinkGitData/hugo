---
description: 根據提供的內容或草稿，產生統一規格的 Hugo 部落格文章
---

當使用者呼叫此 Workflow (例如輸入 `/create-post` 並附帶一段草稿、筆記或想法) 時，請嚴格執行以下任務：

1. **分析主題**：仔細閱讀使用者提供的內容，提取出核心重點，並為其定出一個專業且具吸引力的「文章主題」。
2. **套用版型生成文章**：遵守下方的【系統寫作規範】來撰寫完整的部落格文章。

### 🤖 系統寫作規範

請以「專業技術部落格作者」的身分，套用以下格式、風格與結構。

**【風格與語氣】**

* **語氣：** 專業、條理清晰、具備吸引力，帶有「科技實驗室/探險者」的調性。
* **排版：** 善用標題、條列式清單、**粗體**標示關鍵字，提升文章的可掃視度。請以「繁體中文」輸出所有內容。
* **Markdown 中文粗體防呆（極度重要）：** Hugo Goldmark 解析器對於粗體語法的空格非常嚴格。**絕對規則**：星號 `**` 必須緊貼著裡面的文字，並且在外側留半形空白。**絕對不可以**在星號內側加空白！
  * ✅ **正確寫法**：`這是一段 **重要的** 中文。` 或 `我們邁入 **「代理人時代」** 領域。` (星號緊貼文字，外側留空白)
  * ❌ **錯誤寫法 (吃星號破版)**：`這是一段** 重要的 **中文。` 或 `我們邁入 ** 「代理人時代」 ** 領域。` (星號內側有空白，會導致渲染失敗露出實體星號！)
* **資安與去識別化 (極度重要)：** 文章內文與圖表中，**絕對禁止出現公司特有識別字眼** (例如 `colatour`)。請一律將其改寫為通用代稱 (例如 `enterprise`, `example.com`, `company` 等)，確保不外洩機密資訊。
* **視覺引導（非常重要）：** 在 H2 標題 (`##`) 前面必須加上適合的 Emoji（例如：🌟、📋、🏗️、💻、🚀、🤖 等）。善用引言區塊 (`>`) 來標示重點提示 (`> 💡 **經驗分享**：`) 、程式碼提示 (`> 🤖 **Agent Prompt**:`) 或 Live Demo 連結。

**【Frontmatter 結構】**
最開頭必須嚴格包含以下 YAML Frontmatter：

```yaml
---
title: "[為文章產生的專業性標題]"
date: [請填寫當前時間，格式必須為 ISO 8601，如：2026-04-08T12:00:00+08:00]
draft: false
tags: ["[根據內容萃取的標籤1]", "[標籤2]"]
categories: ["[適當的分類，例如：Tech, AI Projects, Web Development 之一]"]
mermaid: true # 若內容會產生架構流程圖請設為 true，否則省略此欄位
cover:
  image: "images/[與主題相關英文短名稱]-cover.png?v=1"
  alt: "[簡短的英文圖片替代文字]"
  caption: "[簡短的圖片說明]"
  relative: false
---
```

**【必須包含的預期文章結構】**

1. **前言/導言**：簡要介紹技術核心或該篇筆記解決了什麼痛點。
    * **官方資料引用（極度重要）：** 如果使用者有提供來源網址（如 GitHub Gist、官方文件等），請務必在文章開頭（前言之前），使用引言區塊 (`>`) 加上連結來明確標示資料來源。例如：`> 🔗 **官方資料來源**：[來源名稱](網址)`。
2. **## 📋 目錄**：利用 Markdown 與錨點連結建立目錄清單。
3. **本文段落 (H2)**：根據使用者的內容展開具體的技術章節，請為標題配上對應的 Emoji。
    * 若可將概念圖解，請 **強制使用 Hugo 短代碼 `{{< mermaid >}}` 與 `{{< /mermaid >}}`** 來繪製架構圖/資料流圖。藉由短代碼，可以確保系統載入確保深色模式 (Dark Mode) 常保白底對比的高清容器。**嚴禁使用標準的 Markdown Code Block (**` ```mermaid `**)** 繪製流程圖。
    * **Mermaid 語法防呆（極度重要）：** 由於舊版 Mermaid.js 不支援雙引號標籤，為了避免發生 Syntax Error 導致圖表退化成純文字，**圖表中的所有節點與子圖（subgraph）標籤，絕對禁止使用任何特殊字元（如括號 `()`、斜線 `/`、`&` 等），請一律使用純中英文與數字，並且「不要」使用雙引號包覆**。例如：請寫 `A1[節點名稱與說明]`，絕對不可寫成 `A1["節點名稱 (說明)"]`。
    * 若有程式碼，必須使用有語言標籤 (如 `python` 或 `bash`) 的 Code Block。
4. **## 🚀 結論與未來進階**：統整全篇觀念並給予下一步的行動建議。

**【封面圖片生成與自動裁切 (極度重要)】**
請**主動**使用您的圖片生成工具 (如 `generate_image`)，根據文章主題直接替這篇文章產生一張精美的封面圖片。

* **風格需求：** High quality, tech vibe, subtle anime or modern flat illustration style, clean UI aesthetic, vibrant colors.
* **絕對禁止：** 封面圖片的 Prompt **絕對不可以**寫在最終輸出的 Markdown 文章內容中。請在背景自行構思並產出圖片即可，保持文章結尾乾淨。
* **特殊必行操作：** 產生圖片後，必須立刻運用 Python (`Pillow` / `PIL`) 將該獲得的圖片執行「置中、等比縮放並裁切」至 **`1024x256`** 的長條型比例。**注意路徑防呆**：必須儲存至 Hugo 網站的真實根目錄下（例如 `link-blog/static/images/`，而非專案最外層的 `/static/images/`），以符合首頁卡片格式，保證圖片不會出現 404 破圖。

### 3. 主動協助保存

文章生成並展示給使用者確認後，主動詢問使用者：「是否需要我將這篇文章直接寫入專案的 `link-blog/content/posts/[檔案名稱].md` 中？」。
如果使用者同意，請呼叫 `write_to_file` 工具建立該檔案 (檔名須為英文小寫連字號格式)。

### 4. 部署後自動化檢驗與自我修復 (Post-Deployment Verification)

當文章寫入檔案、Commit 並推播 (Push) 到遠端以觸發部署後，請務必主動執行以下檢驗流程：

1. **取得網址**：推導或向使用者確認該篇文章在測試環境或正式環境的 URL。
2. **啟動瀏覽器檢驗**：使用您的 `browser_subagent` 或 `gstack` 技能開啟該網址，模擬真實使用者閱讀體驗。
3. **執行視覺與結構稽核**：
   * **檢查「吃星號」破版**：尋找畫面上是否殘留實體的 `**` 符號（代表中文字或標點符號旁的空白未被徹底執行，導致粗體失效）。
   * **檢查圖表渲染**：確認 Mermaid 圖表是否成功渲染為圖形。若只看見一堆擠在一起的程式碼，代表語法中誤用了舊版不支援的符號（如括號或引號）。
   * **檢查圖片 404**：確認封面圖案與內文圖片是否破圖。
4. **自動修復與規則進化（Self-Healing）**：
   * 若發現上述任何錯誤，請**主動**修改 Markdown 原始檔修復問題並重新 Push。
   * **進化 Workflow（極度重要）**：針對此次犯下的特定錯誤，您必須自動開啟並編輯本文件 (`.agent/workflows/create-post.md`)，將導致錯誤的特例補充進上方的「防呆規則」中，確保未來的產出不再重蹈覆轍。
